## PA (ประเมินผลงาน) — กติกาที่ตกลงกันไว้ (`pa-my.html`)
- **"บันทึก PA แทนบุคคลในแผนก"**: หัวหน้าเลือกบุคคลในแผนกจากลิสต์ (แสดง % ความคืบหน้ารวม + breakdown ตาม 3 ประเภท PA ต่อคน) แล้วเข้าไปบันทึกด้วยหน้าจอชุดเดียวกับ "ตั้งต้น PA" ของตนเอง
- **"PA ลูกทีม"** (มุมมองหัวหน้าทีม) ถูกจำกัดสิทธิ์ 3 ข้อเสมอ:
  1. แสดงเฉพาะลูกทีมของตนเอง (ไม่มี dropdown เลือกแผนกแบบแอดมิน)
  2. แสดงเฉพาะคะแนน "ลูกทีมประเมินตนเอง" กับ "ท่านประเมิน" เท่านั้น — **ห้ามดึงคะแนนคณะกรรมการมาแสดง**
  3. **ห้ามแสดงคะแนนสรุป/คะแนนรวม** ของลูกทีม (หัวหน้าทีมไม่มีสิทธิ์เห็น) — ต้องมีข้อความอธิบายเงื่อนไขนี้ในหน้าจอเอง ไม่ใช่แค่ comment ในโค้ด กันสื่อสารผิดกับทีมพัฒนา
  - รายละเอียดรายการ PA ลูกทีม: แยกกล่องตาม "การประเมินครั้งที่ N" ชัดเจน, ใช้ไอคอนต่างกันระบุที่มาคะแนน (ลูกทีมประเมินเอง vs หัวหน้าประเมิน), แบ่ง 2 แท็บ "ประเมินผล" กับ "รายงานความก้าวหน้า" (ห้ามแสดงข้อมูลซ้ำซ้อนระหว่าง 2 แท็บ), คลิกทั้งรายการเข้าดูรายละเอียดเต็ม (ไม่ใช้ modal)
- **เมนู "พิจารณา/ประเมินผลงาน"** แยกเป็น 2 ฟังก์ชัน (ไม่มี "ประเมินผลงานตนเอง" ซ้ำกับที่อื่น):
  - **ประเมินผลงานลูกทีม**: ให้คะแนน 1-5 + ความเห็นประกอบทีละรายการ
  - **อนุมัติ PA ของลูกทีม**: ดูรายละเอียดแล้วอนุมัติ/ตีกลับ — **อนุมัติได้พร้อมกันหลายรายการ (bulk) ถ้ามาจากคำขอชุดเดียวกัน แต่ตีกลับต้องทำทีละรายการเท่านั้น และต้องกรอกเหตุผลก่อนยืนยันเสมอ**
- **กฎเรื่องการขออนุมัติ**: รายการ PA ประเภท "อยู่ในแผนธุรกิจ" (ดึงข้อมูลอัตโนมัติจากแผนธุรกิจที่อนุมัติแล้ว) **ไม่ต้องยื่นขออนุมัติซ้ำ** — ให้ตัดออกจากทุกคิว/ตัวนับที่เกี่ยวกับการอนุมัติทั่วทั้งแอป (badge แสดง "ไม่ต้องขออนุมัติ" แทน)
- กรอบ "ช่วงจัดทำ PA" (กำหนดส่ง+ปุ่มส่งขออนุมัติ) และ "ขั้นตอนการขออนุมัติ" (stepper) **แสดงเฉพาะช่วงส่งขออนุมัติเท่านั้น** (`status === 'pending_approval'`) — ไม่ใช่แสดงตลอดเวลา ต้องมีกรอบ DEV note ในหน้าจอบอกเงื่อนไขนี้ชัดเจนจนกว่าจะต่อ backend จริง

## แผนธุรกิจ (Business Plan) — กติกาที่ตกลงกันไว้
- โครงสร้างลำดับชั้นจริง 4 ระดับเสมอ: **พันธกิจ → เป้าหมาย → กลยุทธ์ → แผนงาน**
- หน้า **"แผนธุรกิจทั้งหมดขององค์กร"** (`business-plan-org-report.html`) ใช้ **การ์ดแบนคลิกเข้าดูทีละชั้น (breadcrumb)** — ไม่ใช่ tree ยืดหด/ไม่ใช่ drilldown แบบ expand ซ้อนกัน (ผู้ใช้ไม่ชอบทั้ง 2 แบบที่เคยลองมาก่อน)
- คงช่องค้นหา "ค้นหาแผนธุรกิจหรือผู้รับผิดชอบ" ไว้เสมอในหน้านี้ (ผู้ใช้ชอบรูปแบบเดิม)
- **ตัวเลือกรอบเป้าหมาย (period/round selector)** ต้องเป็นชุดเดียวกัน หน้าตาเดียวกันทุกหน้าจอที่เกี่ยวกับแผนธุรกิจ (`business-plan-responsible-list.html`, `business-plan-cockpit.html`, `business-plan-org-report.html`) — ปุ่มไม่ต้องมี label คำว่า "รอบการวัดเป้า" กำกับ (แสดงแค่ค่าที่เลือกอยู่ก็พอ)
- ชื่อเมนู: **"แผนธุรกิจที่รับผิดชอบ"** และ **"แผนธุรกิจทั้งหมดขององค์กร"** (ไม่มีคำว่า "รายงาน" นำหน้า) — ใช้เหมือนกันทุกหน้า
- รายการ PA ที่ผูกกับแผนธุรกิจ (ประเภท auto) ไม่ต้องขออนุมัติซ้ำ (อ้างอิงกฎ PA ด้านบน)

## กฎความสูงปุ่ม Action (บังคับใช้เคร่งครัดทุกครั้งที่สร้าง/แก้ไข)
- ปุ่ม action "หลัก" ที่อยู่ในหัวหน้า/หัว modal (กลับ, บันทึก, ยกเลิก, ปิด, เพิ่มรายการใหม่ ฯลฯ) ใช้ **height: 40px แบบ fixed เสมอ** — ห้ามใช้ padding แนวตั้งกำหนดความสูงเอง (ขนาดตัวอักษร/ไอคอนต่างกันจะทำให้สูงไม่เท่ากันระหว่างหน้า) จัดกึ่งกลางเนื้อหาด้วย display:inline-flex + alignItems:center + justifyContent:center + padding แนวนอนอย่างเดียว (เช่น padding:'0 18px')
- ปุ่ม action "รอง/ภายในแถว" (เช่น บันทึก/ยกเลิกของแถวย่อยในตาราง, +เพิ่มรายการ, ตั้งเป้าหมายในตาราง, ไปชั้นถัดไป) ใช้ **height: 32px แบบ fixed เสมอ** ด้วยหลักการเดียวกัน
- ปุ่มไอคอนวงกลม (แก้ไข/ลบ แบบวงกลมไอคอนล้วน) ไม่อยู่ในกฎนี้ ใช้ระบบขนาดวงกลมตามบริบทเดิม (28–36px)

## กฎลำดับปุ่ม Action ในฟอร์ม/Modal (บังคับใช้เคร่งครัดทุกครั้งที่สร้าง/แก้ไข)
- ลำดับซ้าย→ขวาคงที่เสมอ: **ยกเลิก/ปิด → ลบ (ถ้ามี) → บันทึก** — บันทึกอยู่ขวาสุดเสมอ (คือ action หลัก/primary), ยกเลิก/ปิดอยู่ซ้ายสุดเสมอ, ลบ (ถ้ามีในกลุ่มเดียวกัน) อยู่ตรงกลาง
- ฟอร์มไหนไม่มีปุ่มไหน ก็ไม่ต้องแสดงปุ่มนั้น แต่ที่เหลือต้องเรียงตามลำดับนี้เสมอ (เช่น มีแค่ ยกเลิก+บันทึก ก็เรียง ยกเลิกซ้าย บันทึกขวา)
- **ทุกปุ่มต้องมีไอคอนกำกับเสมอ** ไม่ใช่ปุ่มนี้มีไอคอน ปุ่มนั้นไม่มี — ไอคอนมาตรฐาน: ยกเลิก/ปิด = `bi-x-lg`, ลบ = `bi-trash`, บันทึก = `bi-check-lg`

## กฎปุ่ม Action (บันทึก/ยกเลิก/ปิด ฯลฯ) — บังคับใช้เคร่งครัดทุกครั้งที่สร้าง/แก้ไข
- **border-radius: 11px แบบ fixed เสมอ** สำหรับปุ่ม action หลักทุกที่ (บันทึก, ยกเลิก, ปิด, กลับ, เพิ่มรายการใหม่ ฯลฯ) ไม่ว่าจะอยู่ใน header ของฟอร์ม, modal, หรือ inline — ห้ามใช้ full pill (99/50%) กับปุ่มเหล่านี้ แม้ในหน้าที่ดูเหมือน modal ลอย
- ปุ่มที่เป็น "ตัวเลือกแบบ radio/segmented" (เช่น เชิงปริมาณ/เชิงคุณภาพ, มาตรฐาน/ตามระดับคะแนน) ไม่ถือเป็นปุ่ม action — อนุญาตให้เป็น pill ทรงแคปซูลได้ตามปกติ เพราะสื่อความหมายการเลือกแบบ toggle ไม่ใช่คำสั่งกระทำ

## กฎ Badge/Chip (บังคับใช้เคร่งครัดทุกครั้งที่สร้าง/แก้ไข)
- **border-radius: 8px แบบ fixed เสมอ** — ห้ามใช้ % หรือ 99/pill เต็มรูป เพราะ % จะให้มุมโค้งไม่เท่ากันตามสัดส่วนสูง/กว้างของแต่ละ badge (badge เตี้ยจะกลายเป็นแคปซูลกลม ไม่ตรงตามทรงที่ต้องการ)
- ถ้า badge มีขนาดใหญ่ขึ้นมาก (ไม่ใช่ป้ายเล็กแบบ tag ปกติ) ให้ขยับเป็นค่า fixed ที่ใหญ่ขึ้นตามสเกล (เช่น 10–12px) แทนการใช้ % เพื่อรักษาคาแรกเตอร์มุมเดิมในขนาดที่ใหญ่ขึ้น

## กล่องสลับ เปิดใช้งาน/ปิดใช้งาน (on/off switch แบบ 2 ช่องต่อกัน)
- ใช้ **border-radius: 8px แบบ fixed เสมอ** (เหมือน badge/chip ทั้งระบบ) — **ห้ามใช้ pill/capsule (99)** กับ control นี้ แม้จะดูเหมือนตัวเลือกแบบ segmented ก็ตาม ให้ยึด badge radius เป็นมาตรฐานสำหรับทุกหน้าที่จะออกแบบต่อไป
- โทนสี: เขียวอ่อนเมื่อ "เปิดใช้งาน" ถูกเลือก, แดงอมส้มอ่อนเมื่อ "ปิดใช้งาน" ถูกเลือก (โทนนุ่ม ไม่จัดจ้าน) ฝั่งที่ไม่ถูกเลือกเป็นพื้นขาว/เทาอ่อน

## โทนสีสถานะ (เขียว/เหลือง/แดง)
- อยู่ระหว่างพิจารณาเปลี่ยนโทน (ปัจจุบัน `#ef4444`/`#f5b301`/`#22b96a` ผู้ใช้รู้สึกว่าเข้ม/จัดเกินไปโดยเฉพาะแดง) — มีหน้าเปรียบเทียบตัวเลือกไว้ที่ `color-palette-options.html` (สีทึบ, tint พื้นอ่อน, ไล่สี, glossy 3D, พาสเทลโทนดิน, ฯลฯ) **ยังไม่ได้ตัดสินใจ/ยังไม่ apply เข้าแอปจริง** — ห้ามเปลี่ยนสีในหน้าจอจริงจนกว่าผู้ใช้จะเลือกทิศทางที่ชอบ
